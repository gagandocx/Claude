# Live MT5 trading system for CCT Rectangle Bot

A 10-file module that bridges the backtested CCT Rectangle strategy to live MetaTrader 5 execution. The system polls MT5 every 60 seconds for fresh 4H/15M/1M candle data, runs the existing `CCTRectangleStrategy` class against it, and executes market orders when signals appear and a five-check risk manager approves. Conservative defaults (1% risk, demo-on, 3% daily loss cap) form a sensible barrier between the aggressive backtester and real capital. Trailing stop management mirrors the backtest logic, moving SL once profit exceeds 3R.

**Watch for:** Credentials stored as plaintext Python literals in a committed config file (confirmed). The strategy's `generate_signals()` returns historical signals over the entire data window—without filtering to only the latest bar, the bot will re-execute old signals on every cycle until the dedup list blocks them (confirmed). `record_trade()` is never called, so the RiskManager's daily P&L and trade-count checks operate on stale data (confirmed). Lot-size calculation ignores `symbol_trade_tick_value` from the broker, defaulting to 1.0, which can produce wildly incorrect position sizes on non-USD-denominated accounts (likely).

## High-level view

The architecture splits cleanly into connector, data feed, risk manager, strategy adapter, and dashboard. Each module has a single responsibility and communicates through dictionaries and dataclasses. The polling loop in `mt5_trader.py` orchestrates them sequentially with a simple `time.sleep` remainder pattern.

The risk manager gates every trade through five checks—daily loss, trade count, drawdown, hours, concurrent positions—but the daily loss and trade-count checks rely on `record_trade()` being called externally, which the main loop never does. The risk manager will therefore only halt on drawdown (equity-based, reads from MT5) or hours violations, never on daily-loss or trade-count limits.

Signal deduplication uses a string ID composed of entry time, direction, and price. Because `generate_signals()` replays the entire historical window each cycle, every previously-generated signal appears in the list. The dedup list prevents re-execution, but only after the signal was executed once—not before. If a signal's price shifted due to floating-point differences across bars, or if the strategy generates a fresh signal at the same level on a new bar, the dedup could miss it.

Credentials sit in `mt5_config.py` as Python constants, checked directly into git. For a production trading bot this is the expected pattern to change before deployment; the README guides users to edit the file, but `.gitignore` or environment variable injection would prevent accidental pushes of real passwords.

<summary>Issues (7)</summary>

1. **`record_trade()` never called** — The main loop executes trades and manages positions but never calls `risk_manager.record_trade()` when a trade closes. Daily P&L and trade-count limits are therefore inert. Add a position-close detection step that reconciles MT5 closed positions and feeds results to the risk manager.
2. **Stale signal execution window** — `generate_signals()` returns all historical setups across the data window. Without filtering to signals whose `entry_time` falls within the last polling interval, the bot attempts to execute old signals on its first cycle. Add a recency filter (e.g., only act on signals from the last 2-3 bars of the entry timeframe).
3. **Plaintext credentials in source** — `mt5_config.py` contains account number and password as literals. Add a `.gitignore` entry for this file or load credentials from environment variables.
4. **Lot-size calculation uses default tick value** — `calculate_lot_size` accepts `symbol_trade_tick_value` but the caller never passes it (defaults to 1.0). For XAUUSD with a real tick value, this may under- or over-size positions. Pull `symbol_info.trade_tick_value` from the connector and pass it through.
5. **Trailing stop uses stale SL as risk reference** — Once a trailing stop modifies SL, the `risk` calculation on the next cycle uses the *new* SL, shrinking the apparent risk and potentially triggering further SL tightening faster than intended. Store the original SL (or initial risk) separately.
6. **`sys.path` manipulation for strategy import** — Inserting the parent directory into `sys.path` twice (once unconditionally, once in the except branch) makes the import order fragile and can shadow unrelated modules named `config` or `utils`. Consider a proper package structure or explicit path qualification.
7. **No closed-position reconciliation** — The bot tracks open positions but has no mechanism to detect when MT5 closes a position (via SL/TP hit or manual close). Without this, `record_trade()` can never be called, and the dashboard's daily stats will always show zero trades.

## Signal lifecycle and deduplication gap

The strategy class (`CCTRectangleStrategy.generate_signals()`) is designed for batch backtesting: it receives a full DataFrame window and returns every trade setup found within it. In a live context, this means each 60-second cycle produces a list of all historical signals from the last 200 4H bars, 500 15M bars, and 500 1M bars. The live trader takes `signals[-1]` and checks whether its ID has already been executed:

```python
signal_id = (
    f"{latest_signal.entry_time}_{latest_signal.direction}_"
    f"{latest_signal.entry_price:.5f}"
)
if signal_id in self._executed_signals:
    return
```

On the very first cycle after startup, this list is empty, so whatever the latest historical signal is (possibly from hours ago) will be sent to MT5 for execution. Even if market conditions have moved well past that entry, the order fires at market price with the signal's original SL/TP—potentially with a stop already breached.

A recency gate—discarding signals older than N entry-timeframe bars—would prevent stale execution without requiring changes to the strategy class itself.

## Risk manager disconnect from trade outcomes

`RiskManager.can_trade()` checks `_daily_trade_count` and `_daily_pnl`, both of which are only updated by `record_trade()`. The main loop in `mt5_trader.py` never calls this method. The execution path after `send_order()` returns success immediately appends the signal ID and updates the dashboard, but does not register the trade with the risk manager.

The consequence: after the first trade, `_daily_trade_count` remains 0, and the bot would happily execute up to `MAX_TRADES_PER_DAY` trades without ever incrementing the counter. Only the `MAX_CONCURRENT_TRADES = 1` check (which reads live positions from MT5) prevents runaway trading.

Fixing this requires either (a) calling `record_trade()` immediately on execution (with P&L = 0, since the outcome isn't known yet) to at least track counts, or (b) adding a reconciliation step that polls `mt5.history_deals_get()` for recently closed positions and feeds their outcomes back.

## Lot-size calculation and tick-value assumption

```python
lot_size = self.risk_manager.calculate_lot_size(
    account_equity=account_info["equity"],
    stop_loss_distance=sl_distance,
    symbol_point=symbol_info.get("point", 0.01),
)
```

The `symbol_trade_tick_value` parameter defaults to 1.0 inside `calculate_lot_size`. For XAUUSD on most brokers, the actual tick value per standard lot is approximately $1 per 0.01 price movement (i.e., $100 per point for 1 lot). With the default of 1.0 and a point of 0.01, the formula produces `risk_amount / (sl_points * 1.0)`, which may be orders of magnitude off depending on the broker's contract specification. The connector's `get_symbol_info()` does not retrieve `trade_tick_value` from MT5 (the field exists on `mt5.symbol_info()` but isn't extracted), so even if the caller wanted to pass it, the data isn't available.

## Trailing stop risk drift

The trailing stop logic computes `risk = abs(open_price - current_sl)` on each cycle. Once the first trail modification raises (or lowers) the SL, subsequent cycles see a smaller `risk` value. This changes the meaning of `TRAILING_STOP_ACTIVATION_RR` and `TRAILING_STOP_DISTANCE_RR`: they no longer represent multiples of the *original* trade risk, but multiples of the *current* (tighter) distance. The stop will therefore tighten progressively faster as profit grows.

Whether this is a bug depends on intent. In the backtest, the strategy tracks original SL separately. The live trader does not—it uses the position's current SL as reported by MT5.

## Credential exposure

`mt5_config.py` is tracked in git and contains:

```python
MT5_ACCOUNT = 12345678
MT5_PASSWORD = "your_password"
MT5_SERVER = "YourBroker-Demo"
```

The placeholders are harmless, but the file will inevitably be edited with real credentials. Users following the README are instructed to edit this file directly. Without a `.gitignore` entry or an environment-variable loading pattern (`os.environ.get("MT5_PASSWORD")`), real passwords will be committed on the next `git add .`.

## Import fragility via sys.path

```python
sys.path.insert(0, os.path.dirname(os.path.dirname(os.path.abspath(__file__))))
```

This inserts `CCT_Rectangle_Bot/` onto `sys.path`, making *all* of that directory's modules importable by bare name. The strategy imports `config` and `utils`—these are generic names that could collide with system packages or other project files on the path. Additionally, the `config` imported by strategy.py is the backtest config (25% risk, `GC=F` symbol), which is loaded at strategy import time. Its constants don't control the live bot (which uses `mt5_config`), but if the strategy internally references `config.RISK_PER_TRADE` for signal filtering, those thresholds will be the aggressive ones.
