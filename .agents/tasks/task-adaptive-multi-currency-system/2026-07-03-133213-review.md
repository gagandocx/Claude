# Adaptive multi-currency regime-aware trading system (v3)

A 7,400-line adaptive trading system that classifies markets into six regimes, routes to five specialized strategies, sizes positions via Kelly/equity-curve feedback, and manages risk across multiple symbols with portfolio-level circuit breakers. The two critical v2 issues are resolved: `PerformanceRecord` now uses EMA-based `win_rate` (alpha=0.12, ~10 trades to shift preferences), and regime encoding uses deterministic 2-letter codes (TU/TD/RN/RW/VB/MR) that decode unambiguously. The remaining accepted trade-offs (no automated test suite, contract size as user responsibility, mild loss_boost=1.2 on aggressive profile) are bounded and documented.

**Watch for:** Contract size mismatch between hardcoded `SYMBOL_METADATA` and actual broker settings remains unvalidated at runtime -- a broker using micro-lot contract sizes would produce lot sizes 100x wrong (confirmed, accepted as user responsibility). The aggressive profile's `loss_boost=1.2` after 3 consecutive losses is capped at 15% max risk, making the exposure increase modest (~1.8% additional risk at worst) but still directionally wrong after demonstrated failure (confirmed, accepted trade-off).

**Verdict**: APPROVED

## High-level view

The EMA-based win rate in `PerformanceRecord` (alpha=0.12) gives roughly 63% of weight to the most recent 8 trades. After a regime shift where a strategy starts losing, the system needs only ~10-12 trades to halve its confidence in that strategy and switch to the next candidate. This is the core adaptation mechanism and it now works correctly in both live and backtest paths.

The 2-letter regime code mapping (`REGIME_CODES` / `REGIME_FROM_CODE` dicts in `live_trader.py`) eliminates the truncation ambiguity that previously caused all live trade results to collapse into `RANGING_NARROW`. The encoding is injected into the MT5 order comment and decoded deterministically in `_detect_closed_trades()`, closing the feedback loop between execution and learning.

Position sizing operates conservatively for the balanced profile: 8% grow risk, 2% protect base, 10% absolute cap, Kelly as upper bound (half-Kelly), exponential decay with adaptive `dd_power` that steepens as drawdown deepens. A $1000 account risks at most $80 per trade at equity highs, decaying rapidly in drawdown and halting entirely at 20%.

The risk management stack is layered: per-symbol 5% cap, portfolio 15% cap, 6 concurrent positions max, correlation reduction above 0.7, daily 5% loss limit, session blackout hours, and the 20% drawdown circuit breaker.

## EMA-based adaptation speed

The `PerformanceRecord` in `strategy_selector.py` tracks win rate as an exponential moving average of binary win/loss signals:

```python
ema_win_rate: float = 0.5  # Prior: assume 50/50
alpha: float = 0.12        # ~8 trades carry 63% of weight

def update(self, pnl: float):
    win_signal = 1.0 if pnl > 0 else 0.0
    if not self._has_data:
        self.ema_win_rate = win_signal
        self._has_data = True
    else:
        self.ema_win_rate = self.ema_win_rate * (1.0 - self.alpha) + self.alpha * win_signal
```

With alpha=0.12, the half-life is `ln(0.5) / ln(1 - 0.12) ~ 5.4 trades`. After 10 consecutive losses starting from a 0.6 win rate, the EMA drops to `0.6 * 0.88^10 ~ 0.17` -- well below the threshold where `sharpe_estimate` goes negative and triggers strategy switching.

## Regime code round-trip in live trading

The encoding path in `_send_order`:

```python
regime_code = REGIME_CODES.get(regime, regime[:2])
comment = f"Adaptive_{strategy}_{regime_code}"
```

The decoding path in `_detect_closed_trades`:

```python
parts = comment.split("_")
if len(parts) >= 3 and parts[0] == "Adaptive":
    strategy_name = parts[1]
    regime_code = parts[2]
    regime_name = REGIME_FROM_CODE.get(regime_code, "RANGING_NARROW")
```

The `REGIME_CODES` dict maps all six regime names to unique 2-letter codes, and `REGIME_FROM_CODE` inverts it. Total comment length (`Adaptive_ScalpMomentum_VB` = 25 chars) stays within MT5's 31-character limit.

## Kelly interaction with early trading

Before the scorecard accumulates data, the position sizer detects uninformative defaults (`avg_win==1.0 && avg_loss==1.0 && win_rate==0.5`) and falls through to configured Kelly defaults (win_rate=0.45, avg_win=2.0, avg_loss=1.5). This produces `f* = (1.33*0.45 - 0.55)/1.33 = 0.038`, half-Kelly = 0.019. Kelly then acts as upper bound: `min(risk_grow, 0.019) = 0.019`, meaning early trades use ~2% risk regardless of grow/protect mode. This is more conservative than the v1 concern suggested (where Kelly was thought to be disabled entirely).
