# Adaptive multi-currency regime-aware trading system (v2)

Seven of nine v1 issues were addressed: martingale removed (`loss_boost = 1.0` for balanced), max risk capped at 10%, backtester now uses `StrategySelector`, live trader detects closed trades and feeds results to the learner, Hurst weight reduced to 0.15, Kelly defaults produce a positive `f*`, and PerformanceRecord no longer applies asymmetric decay. Two issues remain unaddressed, and the fixes introduce one new concern.

**Watch for:** `PerformanceRecord.win_rate()` uses cumulative non-decayed counters (`wins` and `losses` grow monotonically), which means adaptation to changing market conditions is extremely slow -- the system will take hundreds of trades to shift its strategy preferences (confirmed). Regime name truncation in MT5 comments causes every closed trade to be attributed to `RANGING_NARROW` regardless of actual regime, making the live learning loop ineffective (confirmed). Contract size mismatch between hardcoded `SYMBOL_METADATA` and actual broker settings is still unvalidated (confirmed).

## High-level view

The position sizing fix is clean: `risk_grow = 0.08`, `max_risk_cap = 0.10`, `loss_boost = 1.0`. On a $1000 account this means maximum $80 risk per trade in grow mode. The aggressive profile still sets `loss_boost = 1.2`, creating a mild Martingale after 3 consecutive losses.

The backtester now instantiates `StrategySelector()` and routes regime-to-strategy identically to the live trader, feeding closed-trade PnL back to `strategy_selector.record_result()` so the scorecard evolves during backtest.

The live trader's `_detect_closed_trades()` closes the feedback loop by comparing position snapshots across ticks and feeding results to `_selector.record_result()`. However, the parsing relies on the comment format `"Adaptive_{strategy}_{regime[:8]}"`, which truncates regime names so they never match `MarketRegime.__members__`. Every result falls back to `RANGING_NARROW`, corrupting the scorecard entirely.

The `PerformanceRecord` fix replaced exponential-decayed wins/losses with raw counters. This eliminates the asymmetry bug but makes `win_rate()` non-adaptive: it returns the all-time win rate, so the system cannot respond quickly to changes in strategy effectiveness across regimes.

<details>
<summary>Issues (5)</summary>

1. **Non-adaptive win_rate in PerformanceRecord** -- `wins` and `losses` are raw counters that never decay. After 100+ trades, recent performance changes barely move `win_rate()`. Either apply exponential decay symmetrically to both (decay before incrementing), or track win rate via `ExponentialStats` as done in `StrategyPerformanceTracker`.
2. **Regime name truncation corrupts live scorecard** -- MT5 comment format truncates regime to 8 chars (`regime[:8]`), so `TRENDING_UP` becomes `TRENDING`, fails `MarketRegime.__members__` lookup, and defaults to `RANGING_NARROW`. Every trending/breakout trade result gets attributed to the wrong regime, making learned strategy selection meaningless in live mode.
3. **Contract size unvalidated against broker** -- `SYMBOL_METADATA` contract sizes are hardcoded. The live trader has `mt5.symbol_info()` available during `connect()` but never uses `trade_contract_size` to override the metadata. A mismatch silently produces wrong lot sizes.
4. **No test suite** -- Zero automated tests. Position sizing edge cases, risk manager rejection logic, regime detection thresholds, and the strategy selector's learning loop have no verification beyond the demo.
5. **Aggressive profile retains mild Martingale** -- `loss_boost = 1.2` with 3-loss threshold and 15% cap is bounded, but still increases exposure after demonstrated failure. Document this explicitly as an intentional risk trade-off or remove it.

## PerformanceRecord win_rate stagnation

The v1 fix removed decay from `wins` and `losses`, solving the asymmetry bug where alternating results caused both counters to decay toward zero. But the fix went too far:

```python
def update(self, pnl: float):
    self.trade_count = self.trade_count * (1.0 - self.alpha) + 1.0  # decays
    if pnl > 0:
        self.wins += 1.0       # never decays
    else:
        self.losses += 1.0     # never decays
```

After 200 trades with 50% win rate, a subsequent 20-trade losing streak changes `win_rate()` from `100/200 = 0.50` to `100/220 = 0.45`. The system needs approximately 100 consecutive losses to push win_rate below 0.33 where `sharpe_estimate` goes negative enough to trigger strategy switching. Meanwhile `avg_pnl` and `avg_win`/`avg_loss` do use exponential decay and will respond quickly -- but `win_rate()` drives the `expected_pnl()` calculation and the confidence-weighted selection in `best_strategy_for_regime()`.

The `StrategyPerformanceTracker` in `online_learner.py` solves this correctly: it tracks win rate via `ExponentialStats` (feeding 1.0 for wins and 0.0 for losses), producing a properly recency-weighted estimate. `PerformanceRecord` should use the same approach.

## Regime comment truncation in live feedback

The live trader encodes strategy and regime in the MT5 order comment:

```python
comment = f"Adaptive_{strategy}_{regime[:8]}"
```

Regime names and their 8-char truncations:
- `TRENDING_UP` -> `TRENDING` (8 chars)
- `TRENDING_DOWN` -> `TRENDING` (same!)
- `RANGING_NARROW` -> `RANGING_`
- `RANGING_WIDE` -> `RANGING_` (same!)
- `VOLATILE_BREAKOUT` -> `VOLATILE`
- `MEAN_REVERTING` -> `MEAN_REV`

In `_detect_closed_trades()`:

```python
regime_enum = MarketRegime[regime_name] if regime_name in MarketRegime.__members__ else MarketRegime.RANGING_NARROW
```

None of the truncated names match `MarketRegime.__members__` (which contains `TRENDING_UP`, `TRENDING_DOWN`, etc.), so every closed trade defaults to `RANGING_NARROW`. The scorecard accumulates all live results under the wrong regime, making learned strategy selection meaningless.

Fix: store the full regime name in the comment (MT5 allows up to 31 characters), or encode a numeric index (e.g., `regime.value`).

## Contract size validation gap

During `connect()`, the live trader already retrieves `mt5.symbol_info(symbol)` and logs spread, min_lot, and volume_step. The fix is one line:

```python
# In connect(), after retrieving info:
actual_contract = info.trade_contract_size
# Override the sizer's config if it differs from hardcoded metadata
```

Without this, trading XAUUSD on a broker that uses contract_size=1 (ounces per lot) instead of 100 would produce lot sizes 100x too large.
