# Adaptive multi-currency regime-aware trading system

A new trading system that classifies market conditions into six regimes (trending up/down, narrow/wide range, volatile breakout, mean-reverting), selects from five specialized strategies per regime, and manages position sizing with Kelly criterion, equity-curve feedback, and portfolio-level risk controls. The system targets MT5 live trading on Windows with a backtest engine for validation. It uses only numpy and pandas -- no external ML libraries.

**Watch for:** Martingale-style loss boosting can amplify drawdowns precisely when the system is already wrong (confirmed). Position sizing with 500:1 leverage and 15-25% risk-per-trade can produce lot sizes that risk blowing a $1000 account on a single adverse move (confirmed). The backtest hardcodes `ScalpMomentum` for most regimes, diverging from the `StrategySelector` logic that would govern live trading (confirmed). No test suite exists (confirmed).

## High-level view

The regime detector combines ADX, volatility ratio, EMA slope, and Hurst exponent through a weighted voting system with hysteresis. The approach is sound for a lightweight classifier, though the Hurst exponent calculation on 100 bars with only 3-5 lag points produces noisy estimates that may cause regime misclassification in fast-moving markets.

Position sizing operates in a two-mode system: "grow" mode at equity highs (15% risk by default) and "protect" mode in drawdown with exponential decay. The `loss_boost = 1.8` multiplier after consecutive losses is a soft Martingale -- it increases exposure when the system is demonstrably failing, which conflicts with the stated goal of capital protection.

The backtest engine pre-computes all signals then runs a lightweight bar-by-bar execution loop. This is architecturally clean for speed, but the hardcoded `regime_to_strategy` mapping inside the backtester uses `ScalpMomentum` for trending and breakout regimes rather than the `StrategySelector` class's learned routing. Backtest results therefore don't validate the strategy selection logic that governs live trading.

The live trader has sensible operational design: state persistence via atomic JSON writes, exponential-backoff MT5 reconnection, CSV trade logging, and graceful shutdown. The gap is that closed-trade P&L isn't tracked back to update the `StrategySelector` scorecard or `StrategyPerformanceTracker`, so the online learning components never receive live feedback.

Risk management provides layered protection (drawdown halt, daily loss limit, per-symbol caps, correlation penalty, max positions). The `_compute_trade_risk` formula `(lot * sl_distance * contract_size) / equity` computes dollar risk correctly assuming the units align, but this is never validated against actual MT5 tick values -- a pip_size or contract_size mismatch would silently produce wrong risk figures.

5. **Hurst exponent noise** -- Computing Hurst on 100 bars with ~3-5 lag points yields unreliable estimates. Either increase the window/lags or reduce Hurst's weight in the regime scoring.
6. **No test coverage** -- Zero unit or integration tests for any module. Critical paths (position sizing edge cases, risk manager rejection logic, regime transitions) have no automated verification.
7. **Contract size mismatch unvalidated** -- `SYMBOL_METADATA` contract sizes are hardcoded assumptions. The live trader doesn't cross-check against `mt5.symbol_info().trade_contract_size`, risking incorrect lot sizing.
8. **Kelly criterion with stale priors** -- With default `avg_win = 1.0` and `avg_loss = 1.0`, Kelly computes `f* = (1*0.5 - 0.5)/1 = 0`, which forces `kelly_risk = min_risk_floor`. Kelly is effectively disabled until the scorecard accumulates data, but this isn't documented.
9. **PerformanceRecord.update decay asymmetry** -- When a win arrives, `losses` decays but doesn't update; when a loss arrives, `wins` decays but doesn't update. Over time with alternating results, both `wins` and `losses` decay toward zero, making `trade_count` and `win_rate` unreliable for sparse data.

## Martingale mechanics in position sizing

The `PositionSizer` applies `loss_boost = 1.8` after `streak_threshold = 2` consecutive losses. With a $1000 account in grow mode (15% base risk), two losses followed by a third trade allocate `0.15 * 1.8 = 27%` risk -- exceeding the `max_risk_cap = 0.25` only because the cap clips it. In aggressive profile, `loss_boost = 2.0` with `max_risk_cap = 0.30` means a third-attempt trade risks 30% of equity after the account has already suffered two losses.

The conservative profile sets `loss_boost = 1.0` (disabled), which is the right call. The balanced and aggressive profiles enable it, creating a failure mode where the system doubles down during adverse conditions:

```python
# position_sizer.py line ~210
if consec_losses >= cfg.streak_threshold:
    risk *= cfg.loss_boost  # 1.8x after 2 losses
```

This interacts with the equity curve feedback: if two losses drop equity 5%, the system is in "protect" mode with reduced base risk, but then `loss_boost` partially offsets that reduction. The two mechanisms fight each other.

## Backtest vs live strategy routing

The backtester on lines 260-275 hardcodes:

```python
regime_to_strategy = {
    MarketRegime.TRENDING_UP: "ScalpMomentum",
    MarketRegime.TRENDING_DOWN: "ScalpMomentum",
    MarketRegime.VOLATILE_BREAKOUT: "ScalpMomentum",
    ...
}
```

Meanwhile `SelectorConfig.regime_mapping` defaults to:

```python
MarketRegime.TRENDING_UP: ["TrendFollower", "ScalpMomentum"],
MarketRegime.VOLATILE_BREAKOUT: ["BreakoutTrader", "FadeStrategy"],
```

The live trader uses `StrategySelector.select()` which initially picks the first candidate from the mapping (before learning kicks in). So live trading would start with `TrendFollower` for trending regimes while the backtest validated `ScalpMomentum`. The demo's +41.58% return reflects `ScalpMomentum` performance on synthetic data, not the system that would actually trade live.

## Live trader missing feedback loop

In `_process_symbol`, after `_send_order` succeeds, the trade is logged to CSV. But nowhere in the main loop or any method does the code detect closed positions and feed results back:

```
# What's missing:
# 1. Detect closed positions (compare previous vs current open positions)
# 2. Calculate realized PnL for closed trades
# 3. Call self._selector.record_result(strategy_name, regime, pnl)
# 4. Call self._perf_tracker.record_trade(strategy_name, regime, pnl, duration)
# 5. Update consec_wins / consec_losses
```

Without this, `StrategySelector.scorecard` never updates, so `best_strategy_for_regime()` always returns the first candidate in the mapping. The "online learning" advertised in the README doesn't function in live mode.

## Hurst exponent reliability

The R/S computation uses `lag = int(lag * 1.4) + 1` starting from `min_lag = 10` with `max_lag = 40`. On a 100-bar window this yields approximately lags [10, 15, 22, 32] -- four data points for the log-log regression. With only 4 points and each R/S estimate computed on `n_ret // lag` segments (so 2-6 segments for large lags), the resulting Hurst estimate has high variance.

The regime detector gives Hurst 20% weight for trending detection and 40% weight for mean-reversion detection. At 40% weight, a noisy Hurst estimate swinging between 0.35 and 0.55 can flip the system between MEAN_REVERTING and RANGING classifications on consecutive detections.

## Contract size risk in live trading

`SYMBOL_METADATA` defines `EURUSD` contract size as `100000` and `GBPJPY` as `100000`. The live trader initializes `PositionSizer` with these values:

```python
self._sizers[symbol] = PositionSizer(SizingConfig(
    contract_size=meta.get("contract_size", 100.0),
    ...
))
```

If a broker uses non-standard contract sizes (mini lots = 10000, micro lots = 1000), the sizing formula `lot = (equity * risk) / (sl_distance * contract_size)` would produce lots 10-100x too small or too large. The live trader has access to `mt5.symbol_info().trade_contract_size` but never uses it to override the hardcoded metadata.

## Risk-per-trade dollar exposure

With balanced defaults: `equity = $1000`, `risk_grow = 0.15`, `leverage = 500`, `contract_size = 100` (gold):

```
lot = (1000 * 0.15) / (2.50 * 100) = 0.60 lots
```

A 0.60 lot gold position with SL at 2.50 ($250 ATR distance) risks $150 per trade -- 15% of the account. The aggressive profile with `risk_grow = 0.20` and `max_risk_cap = 0.30` could risk $300 on a single trade from a $1000 account. Combined with `loss_boost = 2.0`, the system can attempt a $300 trade after already losing $200-400.

The `dd_halt = 0.20` (balanced) provides a hard stop at 20% drawdown, but it takes only two max-risk losses to trigger it, at which point the account has lost $200+ permanently.
