# Live MT5 trading bot mirroring the backtest strategy

This change adds a Python live trading bot (`hft_scalper/live_trader.py`) and supporting documentation that executes XAUUSD scalping trades on MetaTrader 5 using the same dual-RSI signal generation, ATR-based stops, and two-mode adaptive position sizing from the parameter-optimized backtest. The bot wraps the backtest's pure-math logic in a class-based MT5 integration layer with auto-reconnection, CSV trade logging, and graceful shutdown.

**Watch for:** Consecutive-win tracking has no mechanism to detect closed trades or attribute wins/losses, so the streak logic will never fire (confirmed). The `max_risk_cap` default is 0.25 in live but the backtest sweep's `.get()` fallback used 0.20 — a silent parameter divergence (confirmed). Equity state resets on restart with no persistence, meaning the bot forgets its peak equity and drawdown position after any interruption (confirmed).

## High-level view

The indicator functions (`compute_rsi`, `compute_atr`) are genuine character-for-character copies. Signal generation and position sizing formulas match. The translation from a vectorized backtest loop to an event-driven live loop introduces gaps in state management: the backtest tracks trade outcomes inline and updates `consec_wins` on every exit, while the live bot has no equivalent trade-outcome detection — making the streak multiplier dead code.

The position lifecycle is the weakest area. The bot counts open positions via `mt5.positions_get` but never inspects closed positions to determine win/loss outcomes. `consec_wins` is permanently stuck at 0, the streak multiplier never activates, and the bot runs a more conservative sizing profile than the backtest would under the same equity curve.

The bar-indexing logic may compute indicators on an incomplete (still-forming) bar rather than the last closed bar, depending on broker-side MT5 semantics. This would introduce signal instability not present in the backtest.

## Trade outcome tracking is missing

The backtest updates `consec_wins` synchronously after every position exit:

```python
if pnl_d > 0:
    consec_wins += 1
else:
    consec_wins = 0
```

The live bot's `_check_closed_trades` method only calls `_get_account_equity()` and `_get_open_position_count()`. There is no call to `mt5.history_deals_get` or `mt5.history_orders_get` to detect closed positions, determine their P&L, or update the win streak counter. The `consec_wins` attribute stays at 0 for the entire runtime.

The streak boost condition (`consec_wins >= streak_n and dd_scale > 0.8`) is unreachable. In the backtest, the streak boost fires frequently during growth phases and is a meaningful contributor to compounding velocity. The live bot operates without this acceleration, creating a behavioral gap that contradicts the "zero discrepancy" claim.

A correct implementation would snapshot the last-known position set, compare it on each bar, identify exits by their absence, then query deal history for those tickets to determine profit/loss.

## Parameter divergence: max_risk_cap

`DEFAULT_PARAMS` in live_trader.py sets `max_risk_cap: 0.25`. The backtest grid hardcodes `"max_risk_cap": 0.25` in the sweep dictionaries, but `run_backtest()` reads the value via `params.get("max_risk_cap", 0.20)` — meaning any parameter set that omits the key falls back to 0.20.

The config loading in live_trader.py (`load_config`) merges loaded JSON over `DEFAULT_PARAMS`, so if the JSON contains a `max_risk_cap` field it'll be used correctly. The risk is the pathway where `aggressive_results.json`'s `best_params` dict doesn't include the key — live would use 0.25 while the backtest that produced those params ran at 0.20.

## Equity state and restart fragility

On startup, `peak_equity` is set to `mt5.account_info().equity`. If the account drew down while the bot was offline (its own SL/TP hits, manual trades, other EAs), the bot initializes with `equity == peak_equity` and immediately enters GROW mode at maximum risk, when the true equity curve is in drawdown territory.

`bar_count` starts at 0 and `last_entry_bars` is initialized to `[-cooldown - 1] * max_positions`. This makes the cooldown check pass immediately, letting the bot open `max_positions` trades on its very first bar after a restart — behavior the backtest never exhibits mid-run.

## Order filling mode hardcoded

```python
"type_filling": mt5.ORDER_FILLING_IOC,
```

Many XAUUSD brokers (especially ECN) require FOK or RETURN modes. The `order_send` returns `TRADE_RETCODE_INVALID_FILL` on those brokers. The bot logs the error but never adapts or tries alternative fill modes. A robust approach would query `symbol_info.filling_mode` bitmask and select a supported mode at connection time.

## Incomplete bar indexing

The bot fetches 500 bars via `mt5.copy_rates_from_pos(symbol, M1, 0, 500)`, computes indicators over all of them, then signals on `bar_idx = len(close) - 1`. In MT5, position 0 is the current (still-forming) bar, so the 500th element is the incomplete bar whose close price changes with every tick.

The `last_bar_time` guard prevents re-evaluation within the same bar, but the first evaluation fires on the bar that just *opened* — not the one that just *closed*. The RSI and close values are unstable at that moment. The backtest always signals on fully-formed bars.

Using `copy_rates_from_pos(symbol, M1, 1, 500)` would skip the forming bar and reference only completed data, matching the backtest semantics.

## No trade exit logging

The CSV captures entries (timestamp, direction, lot, entry price, SL, TP, indicators). There is no mechanism to log when SL/TP fires server-side. For post-hoc comparison of live vs backtest performance, users have no structured exit data — they'd need to manually export MT5's history. Adding periodic deal-history polling (same fix as the `consec_wins` issue) would enable complete trade journaling.
