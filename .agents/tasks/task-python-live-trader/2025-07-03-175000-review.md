This second review pass examines the fixes applied to the five actionable issues from v1. The deal-history-based `consec_wins` tracking, state persistence via JSON, `copy_rates_from_pos` offset fix, fill-mode auto-detection, and exit logging are all in place. ## High-level view

The v1 issues are resolved. `consec_wins` now updates via `mt5.history_deals_get`, filtering for exit deals (`deal.entry == 1`) matching the magic number and summing profit + swap + commission to determine win/loss. The state file uses atomic write-then-rename, persisting peak equity, consecutive wins, bar count, cooldown timestamps, and the deal-time watermark. Fill mode queries the `symbol_info.filling_mode` bitmask with a FOK → IOC → RETURN fallback. Bar fetching uses offset 1, skipping the forming bar.

One minor residual concern: the deal-time deduplication operates at second granularity — if MT5 reports two exits at the same `deal.time` integer (possible for simultaneous SL fills on both position slots), the second deal is skipped on subsequent polling cycles.

## Deal-time deduplication edge case

The deal history loop filters processed deals via:

```python
if deal.time <= self._last_deal_time:
    continue
```

`deal.time` in MT5 is an integer (seconds since epoch). When the bot holds two positions and both hit stop-loss within the same one-second server tick (common during high-volatility spikes on XAUUSD), both deals carry the same `deal.time` value. Within a single call to `_check_closed_trades`, the loop processes all deals in the response before updating `_last_deal_time` at the end of each iteration — so if both deals appear in the same API response, both are processed. The issue only manifests if the API returns them in separate calls (next bar's poll), at which point the `<=` filter skips the second deal because its time equals the updated `_last_deal_time`.

A ticket-based tracking set (persisted alongside `_last_deal_time`) would eliminate this edge case entirely.
