# HFT Scalping Ensemble EA for XAUUSD 1-Minute Bars

A complete quantitative trading pipeline: load 6M broker ticks, profile market microstructure, backtest five strategy variants, run walk-forward optimization, then produce a consensus-driven MQL5 EA that combines Order Flow Imbalance, Multi-Timeframe Momentum, and Spread Fade into an ensemble scoring system. The ensemble trades only when at least 2 of 3 sub-strategies agree on direction, yielding 80% win rate with -10.5% max drawdown on 1 month of data.

**Watch for:** Out-of-sample validation is barely positive (+$29 on a $1000 account) with only 15 trades total, making statistical significance questionable (confirmed). The backtest engine checks SL/TP against bar high/low without resolving intra-bar ordering ambiguity (confirmed). The EA's spread-fade sub-strategy uses a fundamentally different data source (live `SYMBOL_SPREAD` vs backtest's tick-derived `avg_spread`), creating a backtest-to-live gap (confirmed).

## High-level view

The data analysis module establishes that XAUUSD 1-minute bars exhibit negative short-lag autocorrelation and contrarian order-flow persistence, providing a rational basis for the mean-reversion and OFI-fade strategies. The pipeline converts these statistical properties into five independent strategy implementations, optimizes each via walk-forward grid search, then selects the ensemble as the production winner based on Calmar ratio and validation PnL rather than raw return.

The ensemble's selectivity (15 trades in a month) is both its strength and its vulnerability: drawdown stays at -10.5%, but the sample is far too small for reliable inference about forward performance. The OrderFlow solo variant generated 429 trades and $2,519 PnL, but failed validation (-$220), confirming the optimizer correctly penalized it for overfitting.

The MQL5 EA translates the Python ensemble logic into MetaTrader 5 with daily-loss caps, peak-equity drawdown shutoff, ATR trailing stops, and session filtering. The spread-fade component behaves differently in live execution because it reads the broker's real-time spread integer rather than the per-bar average spread used in backtesting, effectively disabling one of the three ensemble signals.

The session filter in the EA uses `TimeCurrent()` (broker server time) while the backtest operated on UTC timestamps. A broker running UTC+2 or UTC+3 shifts the active window by 2-3 hours relative to what was optimized.

<summary>Issues (6)</summary>

1. **Statistical insignificance of ensemble results** — 15 trades over one month cannot establish edge with confidence. Run the ensemble on at least 3-6 months of data or accept that forward-testing is the real validation.
2. **Intra-bar SL/TP resolution order** — When both SL and TP could be hit within the same bar, the engine checks SL first, biasing results toward stop-outs. Add logic to determine which level was touched first using open-to-high vs open-to-low ordering.
3. **Spread-fade data mismatch between backtest and EA** — Python uses per-bar `avg_spread` from tick data; MQL5 uses `SYMBOL_SPREAD` (current broker spread). Record per-bar spread stats or use high-low range proxy consistently.
4. **No margin-call simulation during open positions** — Margin check happens only at entry. A position moving against the account with 1:500 leverage can breach margin requirements mid-trade without the engine closing it.
5. **Optimizer selects by train PnL, not risk-adjusted metric** — `quick_optimize` passes `metric="total_pnl"` despite the docstring mentioning calmar_ratio. This favors high-variance parameter sets.
6. **EA session filter uses server time without UTC offset** — `TimeCurrent()` returns broker server time, which may differ from UTC. The EA should use `TimeGMT()` or allow a configurable UTC offset.

## Backtest engine: intra-bar SL/TP ambiguity

The bar-by-bar loop in `backtest_engine.py` checks stop loss before take profit for long positions:

```python
if lows[i] <= stop_loss:
    # SL hit
elif highs[i] >= take_profit:
    # TP hit
```

When a bar's range encompasses both SL and TP levels (possible during volatile 1-minute candles on gold at $4700), the engine always assigns the loss. A fair resolution would check whether the bar opened closer to one level than the other, or use the open-high-low-close sequence to approximate touch order. Given gold's fat-tailed kurtosis (76x) reported by the microstructure analysis, wide bars are not rare events.

For the ensemble (average 12-bar duration, ATR-scaled SL/TP), this matters less frequently than for the high-frequency OrderFlow strategy (429 trades), but the bias is systematic.

## Sample size and statistical confidence

The ensemble generated 15 trades on 27,572 bars spanning one month. A binomial test on 12/15 winners gives a p-value of ~0.018 against a 50% null — marginally significant. But the validation split produced only 4 trades (+$29.36), which is statistically meaningless.

The OrderFlow strategy's 429 trades provide much stronger statistical power, but its negative OOS result demonstrates that in-sample PnL with many trades does not guarantee generalizability either. The fundamental issue is that one month of gold data contains perhaps 3-5 truly independent volatility regimes, making any strategy selection a choice among very few observations of regime performance.

## Optimizer metric mismatch

In `optimizer.py`, `quick_optimize` calls `optimize_strategy` with `metric="total_pnl"`:

```python
return optimize_strategy(
    strategy_class, bars, param_grid,
    config=config, metric="total_pnl", max_combinations=150
)
```

The docstring states it optimizes for calmar_ratio, but the actual code maximizes raw PnL. This means the optimizer picks parameters that earned the most dollars in-sample, which for a fixed 0.1 lot size on a $1000 account heavily favors aggressive configurations. The two-stage validation (40% train weight, 60% val weight) partially compensates, but the primary filter still passes high-variance candidates to the validation stage.

## Spread-fade implementation gap between Python and MQL5

The Python `SpreadFadeStrategy` computes spread from the bar's `avg_spread` column (mean of all tick-level bid-ask spreads within each 1-minute bar). The MQL5 EA's `GetSpreadFadeSignal()` reads:

```cpp
spreads[i] = (double)SymbolInfoInteger(g_symbol, SYMBOL_SPREAD) * _Point;
if(spreads[i] < _Point)
    spreads[i] = range * 0.1;  // Fallback approximation
```

This reads the current real-time spread for all lookback bars equally, rather than the historical per-bar spread. The lookback array will contain the same value repeated 31 times, making the median-comparison meaningless. The wide-threshold detection will never trigger unless spread changes between the bar-processing moment and the previous tick. This sub-strategy effectively becomes inactive in the live EA, reducing the ensemble to a two-signal system (OrderFlow + MomentumMTF).

## Session filter: server time vs UTC

The EA uses `TimeCurrent(dt)` to extract the hour for session filtering:

```cpp
int hour = dt.hour;  // Server time - adjust to UTC if needed
```

The backtest filtered on UTC hours from the pandas DatetimeIndex (which was parsed as UTC from `time_msc`). If the broker server runs on UTC+2 or UTC+3 (common for Cyprus/EU brokers), the EA's session window shifts by 2-3 hours, potentially including quiet Asian hours and excluding early London hours that produced profitable signals in backtesting.
